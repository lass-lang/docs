Zone Delimiters - Extra Cases

These test cases focus on zone detection and preamble execution without requiring variable substitution.

valid: no delimiter — pure CSS zone

Without --- on line 1, the entire file is CSS zone. This is backward compatible with plain CSS files.

p {
  color: red;
}
p {
  color: red;
}

valid: empty preamble

Opening and closing --- with nothing between them is valid. The preamble is empty, the CSS zone starts immediately after the closing delimiter.

---
---
p {
  color: red;
}
p {
  color: red;
}

valid: zone detection with preamble execution

The preamble executes as JS, CSS zone passes through. No variable substitution in this test.

---
const x = 'defined but not used'
---
p {
  color: red;
}
p {
  color: red;
}

valid: zone detection preserves CSS exactly

CSS zone content passes through unchanged.

---
const unused = 42
---
.button {
  margin: 10px;
  padding: 5px;
}
.button {
  margin: 10px;
  padding: 5px;
}

valid: multiline preamble executes without error

Multiple JS statements in preamble all execute. CSS passes through.

---
const a = 1
const b = 2
const c = a + b
---
.test {
  width: 100px;
}
.test {
  width: 100px;
}

valid: preamble with comments

JS comments in preamble work correctly.

---
// This is a comment
const x = 1 // inline comment
/* block comment */
---
div {
  display: block;
}
div {
  display: block;
}

valid: extra --- in CSS zone

The second --- closes the preamble. Extra --- in the CSS zone is treated as CSS text (which CSS parsers will error on naturally).

---
const a = 1
---
p { color: red; }
---
p { color: blue; }
p { color: red; }
---
p { color: blue; }

valid: delimiter with tab after dashes

A --- followed by a tab and then comment text is recognized as a delimiter. Any whitespace character after --- starts the comment.

---
const $x = 'blue'
---	comment with tab
p {
  color: $x;
}
p {
  color: blue;
}

valid: delimiter with long comment

A --- can be followed by a long descriptive comment. The entire comment text is stripped and has no effect on output.

---
const $bg = '#f0f0f0'
--- this is a very long comment explaining that the CSS zone below defines the page background
body {
  background: $bg;
}
body {
  background: #f0f0f0;
}

valid: no space after dashes is not delimiter

---nospace (no whitespace after the three dashes) is NOT recognized as a delimiter. Since it's on line 1, the file has no preamble and is treated as pure CSS zone, passing through as-is.

---nospace
p {
  color: red;
}
---nospace
p {
  color: red;
}

Preamble Execution

These test cases verify that preamble code executes when the transpiled module is imported. The CSS output is unchanged — variable substitution is tested separately. The preamble runs as standard JavaScript.

valid: preamble with const variable definition

Preamble code executes. Variables are defined in scope. The CSS zone passes through unchanged (substitution tested separately).

---
const color = "blue"
---
p {
  color: red;
}
p {
  color: red;
}

valid: preamble with let and arithmetic

Complex preamble expressions execute correctly.

---
let size = 10
size = size * 2
const doubled = size
---
.box {
  width: 100px;
}
.box {
  width: 100px;
}

valid: preamble with function definition

Functions defined in preamble are available in scope.

---
function rem(px) {
  return (px / 16) + 'rem'
}
const fontSize = rem(32)
---
h1 {
  font-size: 2rem;
}
h1 {
  font-size: 2rem;
}

valid: preamble with $-prefixed variable substitutes in CSS

$name in CSS zone is substituted with the variable value.

---
const $color = "blue"
---
p {
  color: $color;
}
p {
  color: blue;
}

valid: whitespace-only preamble treated as empty

A preamble with only whitespace is treated as empty.

---
   
---
p {
  color: red;
}
p {
  color: red;
}

valid: preamble with console.log side effect

Side effects like console.log execute when module is imported. The CSS output is unchanged - side effects don't affect output.

---
console.log("Preamble executed!")
const message = "hello"
---
.status {
  display: inline;
}
.status {
  display: inline;
}

valid: preamble with async function definition

Async functions can be defined in preamble.

---
async function fetchData() {
  return "data"
}
---
.container {
  display: block;
}
.container {
  display: block;
}

valid: preamble with import statement syntax

Import statements pass through unchanged in transpiled output. The bundler (Vite) resolves them via its module graph. This test only verifies the import syntax is preserved - actual import resolution happens at runtime in bundler context, not in unit tests.

Note: The CSS output below does NOT depend on the import - it just proves the import syntax passes through without errors during transpilation.

---
const imported = true
---
.button {
  background: #3b82f6;
}
.button {
  background: #3b82f6;
}
