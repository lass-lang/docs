CSS Nesting Passthrough

CSS nesting (per CSS Nesting Module Level 1) is standard CSS. The Lass transpiler doesn't know or care about nesting — it passes through as literal text in the assembled CSS string.

This means nesting resolution happens downstream: the bundler's CSS pipeline (PostCSS, Lightning CSS) or the browser itself flattens the nesting. Lass stays out of the way.

Lass symbols ($name, {{ }}, @(prop), //) work inside nested blocks exactly as they do elsewhere in the CSS zone — the scanner doesn't treat nested CSS any differently.

valid: basic nesting

.parent {
  color: red;
  .child {
    color: blue;
  }
}
.parent {
  color: red;
  .child {
    color: blue;
  }
}

valid: & selector

The CSS & nesting selector passes through. It's CSS, not Lass.

a {
  color: blue;
  &:hover {
    color: lightblue;
  }
}
a {
  color: blue;
  &:hover {
    color: lightblue;
  }
}

valid: deeply nested

Multiple levels of nesting pass through unchanged.

figure {
  margin: 0;
  > figcaption {
    background: rgba(0, 0, 0, 0.5);
    > p {
      font-size: 0.9rem;
    }
  }
}
figure {
  margin: 0;
  > figcaption {
    background: rgba(0, 0, 0, 0.5);
    > p {
      font-size: 0.9rem;
    }
  }
}

valid: nesting with @media

Nested @media inside a rule passes through — this is valid CSS nesting.

.foo {
  display: grid;
  @media (orientation: landscape) {
    grid-auto-flow: column;
  }
}
.foo {
  display: grid;
  @media (orientation: landscape) {
    grid-auto-flow: column;
  }
}

valid: Lass symbols inside nested blocks

$name and {{ }} work the same inside nested CSS as anywhere else. The scanner sees CSS zone text — nesting depth doesn't matter.

---
const $accent = 'coral'
---
.card {
  .header {
    color: $accent;
  }
  .body {
    padding: {{ 8 * 2 }}px;
  }
}
.card {
  .header {
    color: coral;
  }
  .body {
    padding: 16px;
  }
}

valid: // comments inside nested blocks

// comments work inside nested blocks — stripped as usual.

.parent {
  // this disappears
  .child {
    // so does this
    color: red;
  }
}
.parent {
  
  .child {
    
    color: red;
  }
}

valid: malformed nesting passes through unchanged

Like all CSS syntax, malformed nesting is not the transpiler's problem. It passes through as-is — the bundler's CSS pipeline or the browser reports the error.

.parent {
  .child {
    color: red;
}
.parent {
  .child {
    color: red;
}

Note: css-nesting has no invalid: test cases because Lass does not parse or validate CSS nesting. Nesting errors are the downstream CSS pipeline's responsibility.
